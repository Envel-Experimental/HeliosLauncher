# Полный отчет по аудиту безопасности HeliosLauncher

В ходе аудита безопасности был проведен анализ конфигураций окна Electron, работы с процессами (IPC), безопасности зависимостей (через `npm audit`), а также механизмов криптографии и выполнения системных команд.

---

## 1. Конфигурация Electron (`BrowserWindow`)
В файле `app/main/WindowManager.js` были проанализированы настройки безопасности главного окна приложения.
Конфигурация реализована **правильно и безопасно**, следуя современным рекомендациям безопасности Electron:
* `nodeIntegration: false` — запрещает доступ к Node.js API из контекста рендерера (защищает от RCE в случае XSS).
* `contextIsolation: true` — изолирует контексты выполнения JavaScript (предотвращает вмешательство скриптов страницы во внутренние API Electron).
* `sandbox: true` — включает песочницу Chromium на уровне ОС, жестко ограничивая возможности рендерера.
* `webSecurity: true` — включены политики безопасности браузера (CORS и т.д.).

**Статус:** ✅ Отлично. Настройки соответствуют "best practices".

## 2. Зависимости (NPM)
Выполненный ранее `npm audit` показал **0 уязвимостей** в текущих зависимостях. 
Тем не менее, рекомендуется настроить автоматическое сканирование (например, Dependabot или Snyk) в CI/CD пайплайне для предотвращения появления уязвимостей в будущем, так как проект использует тяжелые библиотеки (ONNX и др.).

**Статус:** ✅ Проблем не найдено, но требует постоянного мониторинга.

## 3. Криптография и хранение данных
Ключевые данные (например, токены авторизации) обрабатываются и хранятся безопасно:
* Проект использует официальный Electron API `safeStorage` для шифрования токенов перед их записью на диск. Это гарантирует, что ключи шифрования привязаны к учетной записи пользователя в ОС (через DPAPI на Windows, Keychain на macOS или Secret Service API на Linux).
* В случае недоступности `safeStorage` (что бывает редко), реализован надежный fallback с использованием AES-256-GCM.

**Статус:** ✅ Отлично.

## 4. Выполнение системных команд и инъекции (RCE)
При глубоком сканировании исходного кода на использование опасных функций (`exec`, `spawn`, `eval`) были выявлены следующие **потенциальные риски**:

### ⚠️ Уязвимости внедрения команд (Command Injection) в `FileUtils.js`
В `app/assets/js/core/common/FileUtils.js` обнаружено несколько функций, которые запускают PowerShell-команды через `runCommand('powershell', ...)`.
Пути (`archivePath`, `destDir`, `entryPath`) передаются в скрипт через строковую интерполяцию с одинарными кавычками (`'${archivePath}'`):
```javascript
// Пример из FileUtils.js
const psArgs = ['-NoProfile', '-Command', `Expand-Archive -LiteralPath '${archivePath}' -DestinationPath '${destDir}' -Force`];

// Пример из readFileFromZip
$zip = [System.IO.Compression.ZipFile]::OpenRead('${archivePath}');
$entry = $zip.GetEntry('${entryPath}');
```
**Риск:** Если злоумышленник сможет контролировать имя скачиваемого ZIP-архива или путь его распаковки (например, назвав файл `test'; calc; '.zip`), он сможет выполнить произвольный код в системе от имени пользователя. PowerShell интерпретирует одинарную кавычку как конец строки, что позволяет внедрить и выполнить вредоносный код.

**Рекомендация:** Использовать безопасную передачу аргументов. Вместо конструирования строки PowerShell с переменными, следует передавать переменные через массив аргументов PowerShell (например, с использованием блока `param()` и флага `-args`) или полностью отказаться от вызова PowerShell в пользу Node.js библиотек для работы с ZIP (например, `yauzl` или `adm-zip`).

### ⚠️ Риск инъекции в `JavaGuard.js`
В `app/assets/js/core/java/JavaGuard.js` есть неиспользуемая (или редко используемая) функция `runInstaller(installerPath)`:
```javascript
exec(`msiexec /i "${installerPath}"`, (err) => { ... })
```
**Риск:** Аналогичная проблема — если `installerPath` содержит двойные кавычки (например, путь вида `C:\Downloads\java" /q /c calc.exe "test.msi`), это может привести к выполнению произвольного кода.

**Рекомендация:** Заменить `exec` на `execFile` из модуля `child_process`. `execFile` не запускает командную оболочку, а напрямую передает аргументы процессу, что полностью исключает возможность Command Injection. Пример исправления:
```javascript
const { execFile } = require('child_process');
execFile('msiexec', ['/i', installerPath], (err) => { ... });
```

## 5. Межпроцессное взаимодействие (IPC)
В проекте реализован безопасный мост между `main` и `renderer` (contextBridge в `preloader.js` и обработчики в `IpcRegistry.js` / `FsService.js`). В `renderer-entry.js` также присутствует превентивное отключение `window.eval`, что усложняет эксплуатацию потенциальных XSS.
Важно продолжать следовать принципу наименьших привилегий (Principle of Least Privilege) и не добавлять в IPC функции, способные выполнять произвольные команды от имени ОС, так как это может свести на нет защиту песочницы.

**Статус:** ✅ Приемлемо, защита реализована верно.

Общая архитектура безопасности HeliosLauncher выполнена на высоком уровне. Механизмы Electron сконфигурированы максимально жестко.
