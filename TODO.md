# 🚀 HeliosLauncher Stabilization Roadmap

Этот файл является живым документом, описывающим процесс превращения лаунчера из "зависающего на лого" в стабильный и поддерживаемый продукт.

- [x] **Укрепление моков рендерера**: Замена пустых прокси (`empty.js`) на функциональные полифиллы для `path`, `fs` и `os`.
- [x] **Интеграция IPC Bridge**: Реализация синхронных IPC-вызовов для `fs`.
- [x] **Стабилизация E2E тестов**: Исправлено зависание `beforeAll` и восстановлен полный цикл инициализации.
- [x] **Buffer Polyfill**: Реализована поддержка `Buffer` для проверки электронных подписей в рендерере.

---

## ✅ ЗАВЕРШЕНО (COMPLETED)
- **Modern Build System (esbuild)**: Построена быстрая и надежная система сборки. Вместо 10+ разрозненных JS файлов теперь один `renderer.bundle.js`.
- **Main Process Refactoring**:
    - Унифицированы пути через алиасы (`@core`, `@network`, `@common`).
    - Исправлены `ConfigManager` геттеры, которые ломали инициализацию Main процесса.
    - Удалены "Browserify-зомби" из зависимостей.
- **Renderer "Swiss Knife" Polyfills**:
    - `process`: Полноценный объект с `version`, `cwd`, `env.NODE_ENV` и поддержкой EventEmitter.
    - `path`: Реализованы `join`, `dirname`, `resolve` и др. для работы внутри браузера.
    - `fs`: Прозрачный мост к Main процессу через `HeliosAPI.ipc`. Поддерживает асинхронное и синхронное чтение.
    - `os`: Возвращает реальные данные о системе (CPU, RAM) через IPC.
- **Improved Mock Architecture**: `empty.js` теперь возвращает Прокси-функцию-конструктор, что исправляет ошибку `Class extends value is not a constructor`.
- **Cleanup Discord**: Полностью удален `discordwrapper.js` и зависимость `discord-rpc-patch`.
- **Initial Fixes**: Исправлен путь к языковым файлам и ошибки в `UICore` и `DistributionAPI`.

## ❓ ПОЧЕМУ МЫ ЭТО ДЕЛАЕМ?
1. **Зависание на лого**: Происходило из-за "каскадных крашей" в рендерере. Когда мы убрали `nodeIntegration: true`, код начал падать на вызовах `require('path')` или `process.cwd()`. Мы восстановили эти функции через полифиллы.
2. **60к строк в бандле**: esbuild собирает все зависимости в один файл. Это нормально для современного веба. Мы удалили лишние копии, сократив общий вес проекта.
3. **IPC Bridge**: Мы уходим от прямого доступа рендерера к системе (безопасность). Но чтобы не переписывать 100% кода на асинхронный лад, мы добавили синхронные мосты для критичных мест (чтение конфига/переводов).

## 📋 ИНСТРУКЦИЯ ДЛЯ ПРОДОЛЖЕНИЯ
1. **Build**: Всегда запускать `node esbuild.config.js` перед тестами.
2. **Logs**: При падении в рендерере смотреть стек вызовов в выводе Playwright. Теперь он логгирует полный stack trace.
3. **Mocks**: Если новый модуль в `core` требует Node-зависимость, добавь её в `alias` в `esbuild.config.js`. Если это "тяжелый" модуль, используй `empty.js`, если базовый — добавь метод в соответствующий полифилл.
